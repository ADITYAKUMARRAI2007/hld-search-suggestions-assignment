1. Architecture Diagram And Explanation

The system is designed as a production-style typeahead service where the API path is stateless, read-heavy prefix requests are cached, and search-count writes are durably buffered before batched updates.

Suggestion Read Path

  1. User types a prefix such as iph.
  2. API normalizes the prefix and builds suggest:v1:{version}:{rank}:{prefix}.
  3. Consistent hash ring maps the cache key to a cache node.
  4. Cache hit returns top 10 suggestions immediately.
  5. Cache miss queries the suggestion index, caches the result with TTL, and returns debug metadata.

Search Write Path

  1. User submits a query through POST /search.
  2. API validates and normalizes the query and publishes a search event.
  3. Batch worker aggregates repeated queries every 2 seconds or 500 events.
  4. Worker updates query counts, trend buckets, OpenSearch weights, and invalidates affected cache prefixes.

2. Dataset Source And Loading Instructions

Primary dataset: AmazonQAC, a real query-autocomplete dataset from Amazon Search logs.

  • Dataset URL: https://huggingface.co/datasets/amazon/AmazonQAC
  • License: CDLA-Permissive-2.0
  • Fields used: final_search_term, popularity, search_time, prefixes
  • Requirement coverage: train split is much larger than the required 100,000 queries.

The Java importer accepts AmazonQAC CSV or JSONL export files with these fields:

final_search_term,popularity,search_time,prefixes

Load at least 100,000 unique queries:

mvn -Dmaven.repo.local=.m2/repository spring-boot:run \
  -Dspring-boot.run.arguments="--typeahead.dataset.import-path=/path/to/amazonqac_100k.csv --typeahead.dataset.import-limit=100000"

The repository also includes data/sample-amazonqac-smoke.csv only for quick smoke testing. The official dataset source remains AmazonQAC.

4. Design Choices And Trade-offs

PostgreSQL Source Of Truth

PostgreSQL provides durable upserts, indexing, auditability, and easy inspection. It is not the hot read path; Redis and OpenSearch handle typeahead latency. At very large scale, the counter store could be moved to Cassandra, ScyllaDB, or DynamoDB.

OpenSearch Suggestion Index

OpenSearch completion suggesters provide prefix autocomplete with weighted ranking. The trade-off is eventual consistency because weights are refreshed after batch flushes.

Redis Cache With Consistent Hashing

Prefixes are read-heavy and highly repetitive, so caching top suggestions reduces latency. A Java consistent hash ring with 128 virtual nodes per cache node routes keys. Adding/removing a node remaps only a fraction of keys, unlike modulo hashing.

Trending Scoring

Basic mode uses only historical count. Enhanced mode uses:

score =
  log10(historical_count + 1) * 100000
  + log10(recent_count_1h + 1) * 60000
  + log10(recent_count_24h + 1) * 25000

Recent activity is tracked in 5-minute buckets. Old spikes stop dominating because old buckets fall out of the selected window and hybrid cache TTL is short.

Kafka Batch Writes

Kafka is used as a durable write buffer. The batch worker aggregates repeated queries and performs fewer database writes. This trades immediate consistency for lower write pressure and better reliability than an in-memory-only buffer.
